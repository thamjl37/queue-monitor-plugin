------------------------------------------------------------------------------
KEY THING TO UNDERSTAND FIRST (so you read results correctly)
------------------------------------------------------------------------------
A pipeline's parallel { node('x'){...} } branches are NOT separate Jenkins
"Run"s. They appear as queue/executor PLACEHOLDER SUB-TASKS whose display name
is "part of <job> #<N>" - which is exactly the pattern the collector keys on
(regex: ^part of (.+) #\d+$).

Therefore:
  * Bulk parallel node() blocks strongly drive:
      - Summary cards
      - Label Status table
      - Queue Depth Trend chart
      - Saturation detection
      - Executor Scaling Audit
  * The "Recent Execution Pickups" table fires once PER PIPELINE BUILD
    (RunListener.onStarted), NOT once per branch. To fill the Pickups table
    with volume, run many INDEPENDENT builds (Generator 2), not one big
    parallel build.


==============================================================================
A. ENVIRONMENT & TEST CONFIG
==============================================================================
Jenkins: 2.541.2+ on Java 21.
Install: target/queue-monitor-3.0.0.hpi
  Manage Jenkins -> Plugins -> Advanced -> Deploy Plugin -> restart.

Agents / labels - pick one:
  * MINIMAL (single machine): label the built-in node "linux", give it ~4
    executors. Covers everything EXCEPT scaling (engine never scales built-in).
  * RECOMMENDED: add 1-2 inbound agents labelled "linux" / "windows" with
    small executor counts (e.g. 4 and 2). Enables ALL tests incl. scaling.

Tuning (Manage Jenkins -> System -> Queue Depth Monitor):
  * Poll Interval -> 10s (minimum) so transient queue states get sampled and
    the trend chart accumulates points quickly.
  * Keep Executor Scaling Enabled = true for TC8.
  * Set Max Executors Per Agent above the starting count (e.g. 8) so scale-up
    has headroom.
  * Set Scaling Cooldown -> 30 so you don't wait 5 min between scale events.
  * Leave Build Notifications OFF until TC12.

KEY TIMING RULES:
  * Each branch must sleep long enough to span >=2 poll cycles
    (>=30s at 10s polling; use 90-120s).
  * Fan-out MUST exceed total executors for the label, or you will never see
    queue depth or saturation.


==============================================================================
B. LOAD GENERATORS
==============================================================================

GENERATOR 1 - BULK PARALLEL PIPELINE
(drives Label Status / Summary / Trend / Saturation / Scaling)
Create a Pipeline job named "bulk-parallel" with this script:

    properties([parameters([
      string(name: 'FANOUT', defaultValue: '30'),
      string(name: 'SLEEP',  defaultValue: '120'),
      string(name: 'LABEL',  defaultValue: 'linux')
    ])])

    def n = params.FANOUT.toInteger()
    def branches = [:]
    for (int i = 0; i < n; i++) {
      def idx = i
      branches["b-${idx}"] = {
        node(params.LABEL) {
          echo "branch ${idx} on ${env.NODE_NAME}"
          sleep params.SLEEP.toInteger()
        }
      }
    }
    parallel branches


GENERATOR 2 - BULK INDEPENDENT BUILDS
(drives the Pickups table volume)
First create a freestyle job "sleeper-freestyle":
  - Restrict where it runs -> label: linux
  - Build step: sleep 60   (shell/batch)
Then run this fan-out pipeline to launch 20 independent builds:

    def jobs = [:]
    for (int i = 0; i < 20; i++) {
      def k = i
      jobs["t${k}"] = { build job: 'sleeper-freestyle', wait: true }
    }
    parallel jobs

==============================================================================
E. GOTCHAS (so you don't misread a pass/fail)
==============================================================================
1. Pickups table will NOT show per-branch rows for parallel builds - by design
   (one Run per pipeline). Use Generator 2 for pickup volume. NOT a bug.
2. Parallel pickup queue-wait ~= 0 - the pipeline flyweight starts instantly;
   the interesting per-branch waits do NOT generate PickupEvents.
3. Scaling resource gate reads the CONTROLLER's CPU/RAM, not the agent's
   (ManagementFactory.getOperatingSystemMXBean() in SchedulingEngine). If
   scale-up doesn't fire, it may be controller load failing the threshold.
4. Built-in node NEVER scales, and self-labels are excluded from Label Status
   - always test with a non-"built-in" label.
5. First-run labels may briefly show "built-in" before the job->label hint
   resolves; onCompleted retroactively corrects it.
6. Set sleep long enough - if branches finish inside one poll interval the
   collector may never sample the queued state.
