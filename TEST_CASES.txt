------------------------------------------------------------------------------
KEY THING TO UNDERSTAND FIRST (so you read results correctly)
------------------------------------------------------------------------------
A pipeline's parallel { node('x'){...} } branches are NOT separate Jenkins
"Run"s. They appear as queue/executor PLACEHOLDER SUB-TASKS whose display name
is "part of <job> #<N>" - which is exactly the pattern the collector keys on
(regex: ^part of (.+) #\d+$).

Therefore:
  * Bulk parallel node() blocks strongly drive:
      - Summary cards / Label Status / Trend chart / Saturation / Scaling Audit
  * Pickups, duration anomalies and build NOTIFICATIONS fire once PER BUILD
    (RunListener), NOT once per branch. That is why the suite triggers many
    INDEPENDENT builds of the worker job for those features.


==============================================================================
A. ENVIRONMENT & TEST CONFIG (UAT)
==============================================================================
Jenkins: 2.541.2+ on Java 21. Install target/queue-monitor-*.hpi, restart.

Topology:
  * Controller   - labelled "built-in controller" in the UI. The suite targets
                   it as "built-in" (always matches the built-in node). Give it
                   >= 1 executor so the stress monitor can run there.
  * Linux agent  - label "linux"   (suggest 4 executors for functional tests)
  * Windows agent- label "windows" (suggest 2 executors; curl required - stock
                   on Windows 10 / Server 2019+)
  * Dashboard    - http://devopsdashboard.com (10.100.200.100). Set as the
                   plugin's Build Notification Endpoint URL so every test
                   build's payload is pushed to it (this is how the Jenkins
                   load is DUPLICATED onto the dashboard; dashboard-side
                   verification itself is out of scope for now).

Tuning (Manage Jenkins -> System -> Queue Depth Monitor):
  * Poll Interval -> 10s (and keep the suite's POLL_INTERVAL param in sync).
  * Executor Scaling Enabled = true; Max Executors Per Agent > current count;
    Scaling Cooldown -> 30s.
  * Build Notifications -> enabled, Endpoint URL -> dashboard ingest URL,
    auth (Basic or Bearer) as the dashboard requires.

Suite jobs (one script, two jobs - see test/README.md):
  * queue-monitor-suite  - the orchestrator; choose features via TESTS.
  * queue-monitor-worker - same script; MODE=auto runs it as a worker.
  * First build of each job only registers parameters (exits NOT_BUILT).
  * Worker job must allow concurrent builds (default).

KEY TIMING RULES (the suite enforces/auto-bumps these):
  * Branches must hold >= 3 poll cycles so transient states get sampled.
  * Fan-out must exceed total executors for a label to see queue/saturation.

==============================================================================
D. GOTCHAS (so you don't misread a pass/fail)
==============================================================================
1. Pickups/anomaly/notifications fire once per BUILD, not per parallel branch
   - that is why the worker job exists. NOT a bug.
2. Parallel pickup queue-wait ~= 0 - the pipeline flyweight starts instantly;
   per-branch waits do NOT generate PickupEvents.
3. Scaling resource gate reads the CONTROLLER's CPU/RAM, not the agent's.
4. Built-in node NEVER scales, and self-labels are excluded from Label Status
   - SCALE_LABEL must be a real agent label (linux/windows).
5. First-run labels may briefly show "built-in" before the job->label hint
   resolves; onCompleted retroactively corrects it.
6. Hold times must span >= 3 poll cycles - the suite auto-bumps QUEUE_HOLD,
   but if you shrink other holds below ~30s the collector may miss the state.
7. After the anomaly baseline (5s) is seeded, longer worker builds in pickups/
   stress will also log anomaly ALERTs - expected noise.
8. If jobs live in a folder, set WORKER_JOB to the full path so the pickups
   assertion can match apiPickups' folder-qualified job names.
